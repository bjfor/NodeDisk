# NodeDisk 总体架构设计草案

## 1. 文档目的

这份文档不是替代 1.0 / 2.0 的阶段设计文档，而是站在更高一层，回答下面几个问题：

- NodeDisk 整体到底要做成一个什么系统
- 当前 1.0 / 2.0 在整体架构里分别处于什么位置
- 后续进入分布式阶段时，系统应该分成哪些层
- 哪些模块属于控制面，哪些模块属于数据面
- 后面接入虚拟网络、节点协同、多副本和更强一致性时，应该落在哪里

一句话说，这份文档的目标是：

**给 NodeDisk 提供一个从 2.0 继续往 3.0 演进的总体架构底稿。**

---

## 2. 系统总目标

NodeDisk 的长期目标可以定义为：

**一个面向个人多设备场景的分布式网盘系统。**

它的核心价值不是“做一个网页文件站点”，而是：

- 让多台设备之间可以安全地备份重要文件
- 让文件可以在不同设备之间方便流转
- 让系统能明确知道“哪些内容存在”“哪些设备上有哪些文件”
- 在节点离线、任务失败、部分链路中断时，仍然可以恢复和继续运行

从用户视角看，NodeDisk 最核心的两条产品主线始终是：

1. **大文件备份**
2. **共享文件库 / 跨设备文件流转**

后续所有分布式、同步、控制面、存储层设计，都应该围绕这两条主线服务，而不是反过来为了“分布式而分布式”。

---

## 3. 当前阶段在整体架构中的位置

### 3.1 1.0 解决了什么

1.0 主要解决的是：

- 同步链能不能跑通
- 备份链能不能跑通
- 共享链能不能跑通
- 元数据能不能稳定落库
- mirror / FastDFS 存储后端能不能接起来

1.0 更像是：

**“原型验证阶段”**

它证明了：

- 这套产品目标是可做的
- C++ 技术栈可以支撑核心链路
- 原项目里的上传 / FastDFS / 哈希 / 配置能力可以被新系统吸收

### 3.2 2.0 解决了什么

2.0 主要把系统做成了一个“可持续运行的雏形”：

- 控制面持久化
- 备份历史与恢复闭环
- 共享文件库生命周期
- 审计日志
- 失败重试
- 统一控制面查询
- 统一设备文件视图

2.0 更像是：

**“系统化阶段”**

它解决的是：

- 系统状态能不能长期保存
- 业务能不能被查询、被管理、被恢复
- 用户真正关心的文件流转结果能不能被追踪

---

## 4. 总体架构原则

NodeDisk 后续总体设计建议遵循下面几条原则。

### 4.1 控制面与数据面分离

控制面负责：

- 设备
- 策略
- 任务
- 审计
- 状态查询
- 运维动作

数据面负责：

- 文件扫描
- 文件分块
- 内容摘要
- 传输
- 落盘
- 恢复

这样做的好处是：

- 控制逻辑不会和底层文件传输逻辑搅在一起
- 后面接入更多节点时，控制面可以独立演进
- 后面接 Web / control node / API 也更顺

### 4.2 内容实体与设备文件分离

当前 2.0 已经开始建立这条原则：

- `file_records` 表示内容实体
- `device_file_records` 表示设备上的真实文件

后续必须继续坚持这个方向，因为这直接决定：

- 备份视图是否准确
- 共享接收后是否能纳入设备侧管理
- 恢复后是否能形成设备上下文
- 后面做同步、冲突、设备清理时是否清晰

### 4.3 存储与业务解耦

业务层不应该直接写死某一个存储后端。

应该保持：

- 业务层关心“文件存成功了没有”
- 存储层关心“这个文件实际落到哪里”

所以当前已经存在的抽象是对的：

- `mirror`
- `fastdfs`
- 后面还可以扩 `minio` / 本地对象目录 / 其他对象存储

### 4.4 分布式能力逐层推进

分布式能力建议按下面顺序推进：

1. 多设备接入
2. 节点状态和任务协调
3. 节点间真实网络互联
4. 副本与恢复增强
5. 更强的一致性与冲突处理

不要一开始就直接进入“真正多主一致性”，否则系统复杂度会失控。

## 5. 总体分层

建议把 NodeDisk 的整体架构拆成 5 层。

### 第 1 层：交互与接入层

这一层是系统和人交互的入口。

当前已经有：

- `netdisk_cli`
- `control_node`
- `system_demo`

后续可以扩展：

- 轻量 Web 管理端
- 本地桌面管理工具
- 更稳定的 API 入口

这一层主要负责：

- 发起备份
- 发起共享发布 / 接收
- 发起恢复
- 查看设备、任务、共享、恢复状态

### 第 2 层：控制面

这一层负责系统级管理逻辑。

当前包含：

- `device_manager`
- `policy_manager`
- `task_scheduler`
- `audit_logger`
- `control_plane_service`

它主要负责：

- 节点注册与状态
- 策略管理
- 任务管理
- 审计与运营动作
- 统一查询

后续如果做 `control node` 或 Web 管理端，这一层会成为主要后端聚合层。

### 第 3 层：业务服务层

这一层是最贴近用户功能的层。

当前包含：

- `backup_service`
- `recovery_service`
- `shared_library_service`
- `file_index_service`

它负责：

- 大文件备份
- 恢复
- 共享文件库
- 历史与索引查询

可以把它理解成“控制面之下、传输与存储之上”的业务协调层。

### 第 4 层：数据面核心层

这一层是真正处理文件内容和数据流的地方。

当前包含：

- `sync/chunker`
- `sync/scanner`
- `sync/diff_engine`
- `sync/chunk_reader`
- `sync/sync_writer`
- `network/protocol_codec`
- `network/transport`

它主要负责：

- 文件扫描
- 分块和 hash
- 差异比较
- 文件传输
- 安全写入
- 断点恢复

这一层是未来分布式数据同步的基础。

### 第 5 层：元数据与存储层

这一层负责：

- 元数据持久化
- 文件实体记录
- 设备文件记录
- 存储对象记录
- 实际文件存储

当前包含：

- `metadata_store`
- `storage_backend`
- `fdfs_api`
- `integrity_checker`
- `replica_manager`

当前数据与存储关系是：

- SQLite 管理元数据
- mirror / FastDFS 管理文件内容

## 6. 子系统划分

从总体上，建议将 NodeDisk 长期拆成下面几个子系统。

### 6.1 控制与元数据子系统

职责：

- 节点管理
- 策略管理
- 任务管理
- 审计
- 元数据持久化

当前主要模块：

- `device_manager`
- `policy_manager`
- `task_scheduler`
- `audit_logger`
- `metadata_store`
- `control_plane_service`

### 6.2 备份与恢复子系统

职责：

- 备份执行
- 历史查询
- 恢复请求
- 恢复执行
- 目标设备文件映射

当前主要模块：

- `backup_service`
- `recovery_service`
- `file_index_service`

### 6.3 共享文件库子系统

职责：

- 共享条目创建
- 共享接收
- 共享条目生命周期
- 共享接收状态
- 共享接收后的设备文件映射

当前主要模块：

- `shared_library_service`

### 6.4 同步与传输子系统

职责：

- 文件扫描
- 内容分块
- 同步比较
- 协议编解码
- 断点恢复

当前主要模块：

- `sync/*`
- `network/*`

### 6.5 存储与完整性子系统

职责：

- 对象存储抽象
- FastDFS 对接
- 完整性校验
- 后续副本管理

当前主要模块：

- `storage_backend`
- `fdfs_api`
- `integrity_checker`
- `replica_manager`

### 6.6 分布式网络与节点协同子系统

这是后面 3.0 及以后会逐步增强的重点。

职责：

- 节点发现
- 虚拟网络接入
- 节点间安全互联
- 跨节点任务协同

当前还只是初步形态：

- `node_network_service`
- 设备在线状态判断
- peer 视图刷新

后面会进一步接入虚拟网络环境，比如 ZeroTier 一类方案。

## 7. 关键数据模型

长期看，NodeDisk 的数据模型建议围绕下面几类对象组织。

### 7.1 内容实体

- `file_records`

作用：

- 表示“文件内容是什么”
- 以 `file_id / root_hash` 为核心标识

### 7.2 设备文件

- `device_file_records`

作用：

- 表示“某台设备上有哪些真实文件”
- 这是设备视图、共享接收、恢复落盘的重要基础

### 7.3 存储对象

- `stored_objects`

作用：

- 表示“这个内容实体存到了哪个存储后端”
- 连接内容实体和实际对象存储

### 7.4 备份记录

- `backup_records`

作用：

- 表示“某台设备上的某个文件什么时候做过备份”

### 7.5 共享条目

- `shared_library_entries`
- `shared_library_recipients`

作用：

- 表示共享文件库中的流转对象
- 表示每个目标节点的接收状态

### 7.6 恢复请求

- `restore_requests`

作用：

- 表示某个恢复动作的请求、状态和结果

### 7.7 调度任务

- `scheduled_tasks`

作用：

- 表示备份 / 共享 / 恢复这些系统动作的统一调度视图

## 8. 分布式架构建议

后续进入分布式阶段时，我建议把 NodeDisk 设计成：

**控制面 + 多节点代理 + 存储后端**

也就是：

### 8.1 控制节点

负责：

- 汇总状态
- 统一任务视图
- 策略下发
- 审计
- 运维动作

注意：

当前 `control_node` 更像控制节点雏形，但还不是一个独立常驻服务。

### 8.2 设备节点代理

每台设备上运行一个节点代理，负责：

- 扫描本地文件
- 执行备份
- 执行共享接收
- 执行恢复
- 上报状态

这一步非常关键，因为 NodeDisk 后续真正的“分布式”不是靠一个网页站点完成，而是靠多个设备节点共同工作。

### 8.3 存储后端

存储层可以是：

- 本地 mirror
- FastDFS
- 后续对象存储

它不一定必须和控制节点在同一个进程里，但应该被统一抽象。

### 8.4 网络层

节点之间互联建议分两步：

1. 当前阶段先复用成熟的虚拟网络服务
2. 后续再考虑更深的库级集成

也就是说：

- 先让节点可靠互联
- 再去优化“如何嵌入互联能力”

而不是一开始就把所有底层网络复杂度都引入进来。

---

## 9. 分布式问题的分阶段处理

### 9.1 现阶段重点

现阶段的重点不是“真正多主一致性”，而是：

- 节点互联
- 节点状态感知
- 跨节点任务执行
- 失败重试
- 文件落盘后可追踪

### 9.2 下一阶段重点

当 2.0 收尾后，下一阶段需要重点考虑：

- 节点之间如何真正交换状态
- 哪些任务在控制节点侧触发，哪些在设备侧执行
- 共享和恢复是否要变成真正跨节点调度
- 设备离线时，任务如何排队和补偿

### 9.3 更后面的难点

真正更难的问题会在后面出现：

- 多副本调度
- 节点故障后的副本补偿
- 内容与设备状态的一致性
- 冲突检测
- 更强的一致性模型

这部分建议放在 3.x 以后认真设计，而不要塞进 2.0。

---

## 10. 下一步建议

从这份总体架构草案往下，我建议下一步做两件事：

### 10.1 先把总体架构正式定稿

重点确定：

- 控制节点是不是独立进程 / 服务
- 设备代理是不是后续核心执行单元
- 未来分布式网络层如何接入

### 10.2 再进入分布式设计专题

建议单独开一份文档，专门讨论：

- 节点模型
- 控制节点 / 设备节点关系
- 虚拟网络层
- 任务调度模型
- 副本与故障恢复策略
- 一致性边界

如果说 2.0 的目标是“把系统做成一个雏形”，  
那么下一阶段的目标就是：

**把 NodeDisk 真正设计成一个可扩展的分布式个人网盘系统。**
