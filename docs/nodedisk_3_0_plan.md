# NodeDisk 3.0 规划草案

## 1. 文档目的

这份文档用于回答一个核心问题：

**NodeDisk 在完成 2.0 之后，3.0 到底要做什么，按什么顺序做。**

如果说：

- 1.0 解决的是“主链能不能跑”
- 2.0 解决的是“系统能不能长期运行、被管理、被恢复”

那么 3.0 要解决的就是：

**NodeDisk 能不能真正从“单机系统雏形”走向“多节点协同的分布式个人网盘系统”。**

---

## 2. 3.0 总目标

NodeDisk 3.0 的总目标可以定义为：

**建立控制节点与设备代理协同工作的基本分布式框架，让多设备节点可以在统一控制面下接入、执行任务、同步状态，并为后续副本与更强一致性能力打基础。**

这个目标可以拆成 4 个关键词：

1. **节点化**
   - 不再只是 CLI 和本地流程
   - 每台设备开始具备“节点角色”

2. **协同化**
   - 任务不再只在单进程里触发
   - 控制节点与设备节点开始配合工作

3. **网络化**
   - 节点之间开始通过稳定的虚拟网络互联

4. **分布式基础化**
   - 为后续副本管理和更强一致性预留真正的工程基础

---

## 3. 3.0 不做什么

为了不跑偏，3.0 要明确边界。

3.0 **不建议直接做**：

- 真正去中心化多主一致性
- 元数据共识协议
- 自动冲突合并
- 复杂副本自治修复
- 完整在线 Office/WPS 编辑
- 移动端正式接入
- 大规模多租户设计

这些都重要，但不应该挤进 3.0。

3.0 的目标是：

**把 NodeDisk 做成一个“真正有节点、有控制面、有任务流”的分布式基础系统。**

不是一步做成最终形态。

---

## 4. 3.0 关键结果

3.0 最后应该交付的，不是某一个功能页面，而是下面这些系统结果。

### 4.1 有明确的控制节点

当前 `control_node` 更像一个查询入口。

3.0 应该把它推进成：

- 可常驻运行的控制节点服务
- 负责统一状态、任务和审计
- 能接受设备节点上报
- 能下发任务或供设备节点拉取任务

### 4.2 有明确的设备代理

3.0 应该正式引入 `device_agent` 概念。

它应该负责：

- 设备启动注册
- 设备心跳
- 拉取任务
- 执行备份
- 执行共享接收
- 执行恢复
- 回写结果

### 4.3 有真正的节点间互联

节点之间不能只靠“本地模拟”或同机调试。

3.0 应该具备：

- 多设备加入同一虚拟网络
- 节点有稳定虚拟地址
- 节点之间可以直接进行数据传输

### 4.4 有基本任务调度闭环

至少要实现：

- 控制节点侧创建任务
- 设备代理侧拉取任务
- 设备代理执行任务
- 执行结果回写控制面
- 控制面看到任务完成 / 失败 / 重试

### 4.5 有初步分布式状态模型

这一步不是做强一致性，而是要让系统开始具备：

- 节点视图
- 任务归属
- 目标设备视图
- 设备文件视图
- 失败补偿基础

---

## 5. 3.0 的重点角色

### 5.1 Control Node

3.0 中它应该从“命令入口”升级成“控制服务”。

职责：

- 保存全局控制面状态
- 对外提供统一查询接口
- 管理任务生命周期
- 记录审计日志
- 管理策略与设备

3.0 的关键决策之一就是：

**`control_node` 是否正式常驻化。**

我的建议是：**是。**

理由：

- 后续所有分布式协调都需要一个稳定控制面
- 常驻化之后，Web/API 才有真正的后端载体

### 5.2 Device Agent

这是 3.0 最关键的新角色。

3.0 应该把当前一些分散在 CLI 里的执行动作迁移到 agent。

建议 agent 至少支持：

- `register`
- `heartbeat`
- `poll-task`
- `run-backup`
- `run-share-receive`
- `run-restore`
- `report-result`

### 5.3 Storage Backend

3.0 不一定要新增很多存储后端，但至少要保证：

- `mirror` 继续可用
- `FastDFS` 继续可用
- 控制节点和设备代理都能统一用存储抽象

### 5.4 Virtual Network Layer

3.0 应该明确采用：

**先复用成熟虚拟网络服务。**

也就是：

- 先不自己实现复杂穿透和组网
- 先保证节点稳定互联

---

## 6. 3.0 分阶段实施建议

我建议 3.0 分成 4 个小阶段，不要一次性铺开。

### 6.1 Phase 3.1：控制节点常驻化

目标：

- 把 `control_node` 从查询命令升级成常驻服务雏形

建议做的事：

- 提供本地 HTTP / RPC 查询接口
- 把 `summary / devices / tasks / shared / restores / audit` 固化成服务接口
- 保持当前 SQLite 控制面逻辑不变

这一步的意义：

- 给后续设备代理一个稳定的控制面目标

### 6.2 Phase 3.2：设备代理骨架

目标：

- 引入 `device_agent`

建议做的事：

- agent 启动时注册自己
- agent 定时上报心跳
- agent 能从控制节点拉取任务
- agent 能报告执行结果

这一步的意义：

- 系统真正进入“控制面 + 执行面”模式

### 6.3 Phase 3.3：分布式任务执行

目标：

- 让备份 / 共享接收 / 恢复三类任务从“本地命令触发”升级成“控制节点调度、设备代理执行”

建议做的事：

- 控制节点写任务
- 目标设备代理执行
- 执行结果回写任务状态

这一步的意义：

- NodeDisk 第一次真正跨设备协同工作

### 6.4 Phase 3.4：分布式状态与后续扩展点

目标：

- 为副本和一致性阶段打基础

建议做的事：

- 完善节点状态模型
- 完善设备文件同步状态
- 引入更清晰的副本记录结构
- 明确后续副本补偿入口

---

## 7. 3.0 关键模块规划

建议 3.0 至少新增或强化下面这些模块。

### 7.1 `apps/device_agent`

这是 3.0 最重要的新入口。

作用：

- 每台设备运行的常驻进程

### 7.2 `control_plane/agent_registry`

职责：

- 管理设备代理注册
- 管理心跳
- 管理 agent 状态

### 7.3 `control_plane/task_dispatcher`

职责：

- 统一封装“控制面如何给设备分发任务”

### 7.4 `network/node_client` / `network/node_server`

职责：

- 设备节点与控制节点之间的通信

### 7.5 `security/overlay_adapter`

职责：

- 抽象虚拟网络环境
- 后面接 ZeroTier 等方案时，不让业务层直接依赖具体实现

---

## 8. 3.0 的数据模型变化建议

3.0 不一定要推翻现有元数据模型，但建议增强下面几类对象。

### 8.1 NodeInfo 增强

建议新增：

- `status`
- `agent_version`
- `capabilities`
- `last_error`

### 8.2 ScheduledTaskRecord 增强

建议新增：

- `assigned_node`
- `picked_at_epoch`
- `finished_at_epoch`
- `result_code`

### 8.3 DeviceFileRecord 增强

建议新增：

- `state`
- `last_sync_epoch`
- `version_ref`

### 8.4 ReplicaRecord 正式引入或增强

为后续副本管理做准备。

---

## 9. 3.0 的技术路线建议

### 9.1 当前建议继续保持 C++ 主线

理由：

- 当前核心能力已经都在 C++ 里
- 继续沿现有骨架推进更稳

### 9.2 控制节点先轻量服务化

建议：

- 不必一上来做复杂 Web 框架
- 可以先做轻量 HTTP / RPC 服务

### 9.3 虚拟网络先外部接入

建议：

- 先用现成节点互联环境
- 后面再决定是否深度集成库

### 9.4 存储层继续保持抽象

建议：

- 不要把 agent / control node 和具体存储绑死

---

## 10. 3.0 的验收标准建议

3.0 做完后，至少应该满足：

1. 控制节点能常驻运行
2. 设备代理能注册和心跳
3. 设备代理能从控制节点获取任务
4. 备份任务能由控制节点下发并由设备代理执行
5. 共享接收任务能由控制节点下发并由设备代理执行
6. 恢复任务能由控制节点下发并由设备代理执行
7. 控制节点能统一看到任务状态和节点状态

只要做到这一步，NodeDisk 就已经真正迈进分布式阶段了。

---

## 11. 一句话总结

NodeDisk 3.0 的本质目标不是“继续堆更多功能”，而是：

**把 2.0 的单系统雏形，推进成一个真正由控制节点和设备代理协同工作的分布式个人网盘基础系统。**
