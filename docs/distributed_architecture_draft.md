# NodeDisk 分布式架构设计草案

## 1. 文档目的

这份文档专门讨论 NodeDisk 在进入分布式阶段之后的整体设计问题。

它主要回答 5 个问题：

1. 控制节点和设备代理之间是什么关系
2. 节点之间怎么互联
3. 任务应该怎么调度
4. 文件和副本应该怎么在节点之间流动
5. 一致性边界应该怎么定义

这份文档不是直接给出最终实现细节，而是先把分布式阶段的基本框架定下来，避免后面边开发边改方向。

---

## 2. 分布式阶段的总体目标

进入分布式阶段后，NodeDisk 的目标不再只是“单机可运行”或者“多设备能操作”，而是：

**让多个设备节点在统一控制面下协同工作，稳定完成备份、共享、恢复和后续的副本管理。**

更具体一点，分布式阶段要达到下面这些能力：

- 多个设备节点可以加入同一个系统
- 节点之间可以可靠互联
- 控制面能感知节点状态
- 任务可以从控制面下发到设备节点执行
- 文件可以跨节点流动
- 节点掉线或局部失败后，系统仍能恢复和继续工作

注意：

当前阶段的“分布式”首先应该理解成：

- 多节点
- 有协同
- 有调度
- 有状态

而不是立刻进入：

- 真正多主一致性
- 去中心化元数据共识
- 全自动复杂副本修复

这些更难的能力，应该留到后续更高阶段。

---

## 3. 总体架构建议

我建议 NodeDisk 的分布式架构采用：

**控制面 + 设备代理 + 存储后端 + 虚拟网络**

也就是下面这个基本结构：

```text
                +----------------------+
                |     Control Node     |
                | 状态 / 策略 / 任务 / 审计 |
                +----------+-----------+
                           |
                  control / query / retry
                           |
      -------------------------------------------------
      |                       |                       |
      v                       v                       v
+------------+         +------------+         +------------+
| Device A   |         | Device B   |         | Device C   |
| Agent      |         | Agent      |         | Agent      |
| 扫描/备份/恢复 |         | 扫描/接收/恢复 |         | 扫描/备份/共享 |
+-----+------+         +-----+------+         +-----+------+
      |                      |                      |
      | virtual network      | virtual network      |
      |                      |                      |
      -----------------------------------------------
                           |
                    node-to-node transport
                           |
                 +------------------------+
                 | Storage Backend Layer  |
                 | mirror / FastDFS / ... |
                 +------------------------+
```

这套设计的核心思想是：

- 控制面负责“知道系统发生了什么”
- 设备代理负责“真的去做事情”
- 虚拟网络负责“让节点之间能可靠互联”
- 存储层负责“让文件内容有地方落”

---

## 4. 核心角色定义

### 4.1 Control Node

控制节点不是用来传文件的，它的主要职责是管理系统状态。

职责包括：

- 设备注册与设备状态维护
- 策略管理
- 任务记录与任务重试
- 审计日志
- 统一查询
- 统一运营动作入口

当前 `control_node` 已经是这个角色的早期雏形，但还只是一个命令行入口。  
后续它可以演进成：

- 常驻服务
- 本地 HTTP / RPC 服务
- Web 管理端的后端

### 4.2 Device Agent

设备代理是分布式阶段最关键的执行单元。

每台设备都应该有一个设备代理，负责：

- 扫描本地文件
- 执行备份
- 执行共享接收
- 执行恢复
- 上报设备状态
- 在后续阶段执行节点间同步任务

这意味着后续真正的 NodeDisk 节点不是“网页”或者“控制面命令”，而是：

**每台设备上的一个长期运行的 agent 进程。**

### 4.3 Storage Backend

存储后端负责保存文件内容。

当前已经有：

- `mirror`
- `fastdfs`

后续可以继续扩：

- 本地对象目录
- MinIO
- 其他对象存储

这里要继续保持一条原则：

**存储后端只是文件内容存储层，不承担控制面职责。**

### 4.4 Virtual Network Layer

虚拟网络层负责让节点之间建立稳定可控的私有互联。

建议分两步：

1. 先复用成熟服务
2. 后面再评估是否深度集成库

原因是：

- 当前真正难的地方不在 NAT 穿透本身
- 真正难的是节点模型、任务调度、状态一致性和副本恢复
- 先复用成熟虚拟网络，更有利于尽快推进系统主线

---

## 5. 控制节点与设备代理的关系

这是分布式设计里最重要的问题之一。

我建议 NodeDisk 采用：

**控制面集中，执行面分布**

也就是：

- 控制节点是中心化的
- 设备代理是分布式的

### 为什么这样更适合当前阶段

因为你现在的产品核心是：

- 个人多设备使用
- 备份
- 文件流转
- 强调可靠和可恢复

这类场景下，先让控制面集中，会有几个明显好处：

1. 状态更清楚  
设备、策略、任务、共享、恢复都能统一看。

2. 开发复杂度更低  
不需要一开始就做多主元数据同步。

3. 更容易排错  
出了问题先看控制面，再看设备执行链。

4. 更适合逐步演进  
后面如果真要做更强的一致性，也有清晰基线。

### 设备代理应该向控制面暴露什么

建议设备代理至少周期性上报：

- `node_id`
- `device_name`
- `virtual_ip`
- `last_seen`
- 本地扫描状态
- 当前运行任务
- 最近失败任务
- 设备侧文件统计

控制节点则保存这些信息，并提供统一查看。

---

## 6. 虚拟网络接入建议

### 6.1 当前建议

当前阶段建议优先采用：

**先用成熟虚拟网络服务把节点连起来。**

例如：

- ZeroTier 类方案

理由是：

- 开发重点不应该先放在底层穿透
- 先把业务和控制面跑稳更重要

### 6.2 在系统里的位置

虚拟网络层应该位于：

- 设备代理之下
- 数据传输层之上

也就是说：

- `sync_core` 和未来 agent 之间不直接关心公网穿透
- 只关心“我能不能可靠访问另一个节点”

### 6.3 后续演进

后面如果要更进一步，可以考虑：

- 节点程序自己管理虚拟网络接入
- 把虚拟网络能力从“外部依赖”演进成“系统一部分”

但这一步建议放后面，不要抢当前主线。

---

## 7. 任务调度模型建议

任务调度是分布式阶段的第二个重点。

### 7.1 当前适合的模型

建议采用：

**控制节点负责调度，设备代理负责执行**

也就是：

1. 控制面产生任务
2. 任务写入统一任务表
3. 目标设备代理拉取或接收任务
4. 设备代理执行
5. 设备代理回写结果
6. 控制面汇总状态

### 7.2 任务类型建议

后续至少会有这几类任务：

- `backup`
- `share_publish`
- `share_receive`
- `restore`
- `sync_pull`
- `replica_repair`

当前 2.0 已经覆盖的主要是前四类。

### 7.3 调度方式建议

一开始建议先做：

- 控制面写任务
- 设备代理轮询任务

而不是一开始就做复杂推送。

原因：

- 简单
- 稳定
- 更容易排查问题
- 更适合个人项目持续演进

### 7.4 任务状态建议

任务层至少应统一为：

- `queued`
- `running`
- `completed`
- `failed`

后续再按具体业务扩展子状态。

---

## 8. 文件与副本流动模型

### 8.1 内容实体层

仍然建议继续以内容实体为中心：

- `file_records`

表示“内容是什么”。

### 8.2 设备文件层

继续强化：

- `device_file_records`

表示“哪台设备上有什么文件实例”。

### 8.3 存储对象层

- `stored_objects`

表示“内容实体落到了哪个存储后端”。

### 8.4 未来副本层

进入更强分布式阶段后，需要新增或强化：

- `replica_records`

它表示：

- 哪个内容实体
- 有哪些副本
- 分布在哪些节点或哪些存储位置
- 当前副本状态如何

这样后面才能做：

- 节点丢失后的副本补偿
- 副本数量检查
- 更强的恢复和修复

---

## 9. 一致性边界建议

这里是分布式设计里最容易失控的地方，所以要先定边界。

### 9.1 当前阶段不要追求的东西

当前阶段不建议直接做：

- 真正去中心化多主一致性
- 节点间元数据共识
- 自动冲突合并
- 复杂副本自治修复

因为这些会把系统复杂度一下子拉高。

### 9.2 当前阶段应该先保证什么

当前阶段应该先保证：

1. 单个任务的状态一致  
一个备份/共享/恢复任务，其状态必须清楚。

2. 内容和设备文件映射一致  
文件落盘后，设备文件视图要能反映出来。

3. 失败可见、可重试  
失败不能悄悄吞掉。

4. 控制面能解释当前状态  
系统必须“可观察”。

### 9.3 后续再逐步增强

真正更强的一致性问题，建议按这个顺序推进：

1. 单主控制面 + 多设备执行
2. 副本状态一致
3. 跨节点文件同步状态一致
4. 最后再考虑更复杂的多主场景

---

## 10. 下一阶段建议

从这份分布式草案往下，我建议后续工作分成两块。

### 10.1 架构定稿类工作

先把下面这些定清楚：

- 控制节点是否做成常驻服务
- 设备代理的职责边界
- 虚拟网络接入方式
- 控制面和设备代理之间的任务交互方式

### 10.2 工程推进类工作

然后再落代码：

1. 新增 `device_agent` 角色骨架
2. 把当前 CLI 里的执行动作逐步迁到 agent
3. 让 `control_node` 更像真正控制节点
4. 定义节点间状态同步和任务拉取接口
5. 再进入副本与更强一致性的阶段

---

## 11. 一句话结论

NodeDisk 后续分布式阶段，最合理的总体方向是：

**控制面集中、执行面分布、虚拟网络先复用成熟方案、文件内容与设备文件分层管理、一致性按阶段逐步增强。**

这是当前最稳、最适合继续往下做的路线。
