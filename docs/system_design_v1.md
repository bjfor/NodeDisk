# 个人分布式网盘系统 1.0 开发文档

## 1. 文档目的

这份文档用于说明当前项目 1.0 版本的整体设计，重点回答下面几个问题：

- 这个项目现在到底是什么
- 整个系统应该怎么分层
- 各个子系统、模块、功能分别负责什么
- 当前已经完成了哪些内容
- 原项目中哪些能力被复用了
- 后续应该沿着什么方向继续演进

这份文档尽量用通俗语言表达，方便开发时统一理解，也方便后续继续拆任务。

---

## 2. 项目定位

当前项目的目标，不再是原来那种“网页优先的云盘网站”，而是一个更偏底层的系统：

**一个以文件同步、备份、跨设备传输为核心的个人分布式网盘系统。**

它的主要特点是：

- 以 C/C++ 为核心实现语言
- 弱化前端，先把内核能力做稳
- 优先解决大文件同步、断点恢复和数据安全
- 尽量复用原项目已经成熟的上传、分片、存储能力

当前 1.0 版本不是最终成品，而是整个系统的第一阶段可运行版本。

可以把它理解成：

**原项目存储能力 + 新增同步内核 = 当前 1.0 版本**

---

## 3. 1.0 版本目标

1.0 的目标是先完成一个“最小可运行闭环”，而不是一步做到最终的完整分布式系统。

### 3.0 从用户角度看 1.0 到底要做什么

从开发角度看，我们会讨论子系统、模块、协议、元数据这些概念。
但从用户角度看，整个项目的核心其实只有两个功能：

1. **大文件备份**
用户希望把本地的重要文件安全备份到自己的网盘系统里，尤其是：

- 虚拟机镜像
- 压缩包
- 工程目录归档
- 研究样本
- 体积很大的二进制文件

用户真正关心的是：

- 文件能不能成功备份
- 大文件传输中断后能不能继续
- 备份后的文件会不会损坏
- 后面还能不能恢复回来

2. **共享文件库**
用户希望在自己的多台设备之间传文件，不再依赖微信、QQ、U 盘或手动拷贝。

用户真正关心的是：

- 在设备 A 放进去的文件，设备 B 能不能方便取到
- 文件是不是只对自己的设备可见
- 文件传输后能不能确认完整
- 能不能知道这个文件是从哪台设备发来的

所以，从用户视角看，这个系统不是“一个网站”，而是：

**一个帮用户完成大文件备份和多设备传文件的个人基础设施。**

### 3.1 1.0 要解决的问题

- 两个节点之间如何交换文件信息
- 如何判断本地和远端的文件差异
- 如何只同步缺失或变化的部分
- 如何保证同步过程中不会把半成品文件当成成功结果
- 如果同步中断，如何继续恢复
- 同步成功后，如何把文件交给最终存储后端

### 3.2 1.0 暂时不做的内容

- AI 相关功能
- 原项目里的公共共享广场
- 图床能力
- 排行榜
- 多副本自动调度
- 完整冲突解决
- 移动端接入
- 重前端界面
- 真正去中心化的多主一致性

## 4. 系统总体架构

从大方向上看，当前系统推荐采用：

**中心化控制面 + 分布式数据面**

这是当前阶段最稳、最适合复用原项目能力的一种架构。

### 4.1 控制面

控制面负责：

- 记录元数据
- 管理设备
- 管理同步任务
- 管理备份和中转策略
- 提供最小管理入口

它更像系统的大脑，决定“做什么”。

### 4.2 数据面

数据面负责：

- 扫描文件
- 分块
- 哈希计算
- 节点间通信
- 缺块同步
- 断点恢复
- 安全写入
- 最终存储

它更像系统的执行引擎，决定“怎么做”。

### 4.3 从用户功能到系统结构的映射

为了避免文档只停留在技术分层，这里把“用户能看到的两个核心功能”映射到系统结构。

#### 功能一：大文件备份

这个功能不是靠某一个模块单独完成的，而是由下面几部分组合起来完成：

- 子系统 4：业务应用子系统  
  负责“备份中心”这个业务入口，决定用户想备份哪些目录、哪些文件
- 子系统 1：控制与元数据子系统  
  负责记录备份任务、文件记录、块记录、最终存储位置
- 子系统 2：同步与传输子系统  
  负责扫描文件、分块、比较、传输、断点恢复、原子提交
- 子系统 3：存储与副本子系统  
  负责把同步完成的文件落到最终存储，比如本地镜像或 FastDFS
- 子系统 5：安全与可靠性子系统  
  负责任务恢复、失败重试、校验和日志

也就是说：

**“大文件备份” = 备份业务入口 + 同步内核 + 最终存储 + 恢复保障**

#### 功能二：共享文件库

共享文件库也不是一个单独页面，而是一条完整业务链。

- 子系统 4：业务应用子系统  
  负责“中转共享库”业务入口，决定文件放进共享库还是从共享库取出
- 子系统 1：控制与元数据子系统  
  负责记录共享文件条目、来源设备、接收状态、过期策略
- 子系统 2：同步与传输子系统  
  负责把文件从一个设备安全传到另一个设备
- 子系统 3：存储与副本子系统  
  负责共享文件在中心节点或目标节点上的实际落盘
- 子系统 5：安全与可靠性子系统  
  负责访问控制、完整性校验、异常恢复

也就是说：

**“共享文件库” = 中转业务入口 + 元数据登记 + 节点间传输 + 最终落盘**

## 4.4 用户工作流程

这一节不从代码出发，而是从用户实际操作出发，说明两个核心功能在系统内部是怎么被组合实现的。

### 4.4.1 大文件备份工作流程

用户视角下的流程：

1. 用户在某台设备上指定一个文件或目录，发起备份
2. 系统识别这个文件属于哪台设备、原始路径是什么
3. 系统开始扫描文件并生成摘要
4. 如果文件很大，系统按块处理，而不是一次性整体处理
5. 节点之间交换索引，判断哪些块已经存在、哪些块需要传输
6. 系统只传输缺失部分，传输中断可恢复
7. 接收端先写 `.part` 文件
8. 全部写完后做整文件校验
9. 校验通过后原子提交为正式文件
10. 文件再交给最终存储后端
11. 元数据层记录这次备份结果

系统内部涉及的模块组合：

- 备份中心模块：接收“我要备份这个文件”的意图
- 目录扫描模块：发现文件
- 分块摘要模块：生成 chunk 和 root hash
- 差异比较模块：判断需要同步哪些内容
- 传输协议模块：完成节点间消息交换
- 恢复写入模块：负责 `.part`、校验、原子提交
- 元数据管理模块：保存备份结果
- 存储适配模块：把结果交给 mirror 或 FastDFS

可以把它理解成：

**用户看到的是“备份一个文件”，系统内部做的是“一次安全的分块同步任务”。**

### 4.4.2 共享文件库工作流程

用户视角下的流程：

1. 用户在设备 A 上选择一个文件，放入共享文件库
2. 系统给这个文件登记来源设备、来源路径、时间和状态
3. 文件进入共享库的存储区域
4. 用户在设备 B 上查看共享文件库列表
5. 用户选择某个共享文件并拉取到本地
6. 系统完成差异判断和块传输
7. 文件在设备 B 上安全落盘
8. 元数据层记录“这个文件已经被设备 B 接收”

系统内部涉及的模块组合：

- 中转共享库模块：提供“放入共享库”和“从共享库取出”的业务入口
- 元数据管理模块：记录共享条目、来源设备、接收状态
- 传输协议模块：在节点间传输共享文件
- 恢复写入模块：负责目标设备的安全落盘
- 存储适配模块：决定共享文件实际存到哪里

可以把它理解成：

**用户看到的是“像微信文件传输助手一样发个文件”，系统内部做的是“一次带来源信息的受控同步任务”。**

## 6. 当前 1.0 的实际落点

虽然上面是完整系统设计，但当前已经实际实现的部分主要集中在以下几个位置：

- 子系统 2：同步与传输子系统
- 子系统 3：存储与副本子系统中的存储适配部分
- 子系统 5：安全与可靠性子系统中的恢复和原子提交部分

在当前代码骨架中，其他子系统也已经补进来了，只是完成度还没有同步内核那么深：

- 子系统 1：控制与元数据子系统，已经具备设备、任务、策略的基础服务骨架
- 子系统 4：业务应用子系统，已经具备备份、恢复、共享文件库、文件索引的基础服务骨架
- 子系统 5：安全与可靠性子系统，已经具备认证、审计、节点互联的基础服务骨架
- 子系统 6：交互与接入子系统，当前先由 `sync_v1` 命令行入口承载

也就是说，当前 1.0 还不是完整产品，而是：

**先把同步内核做成系统中的核心组件。**

从用户角度看，可以把当前 1.0 理解成：

- “大文件备份”这条主线，底层核心链路已经基本打通
- “共享文件库”这条主线，目前还主要停留在系统设计和模块规划阶段，还没有完整业务层落地

换句话说：

- 当前 1.0 已经比较接近“备份内核原型”
- 还没有达到“共享文件库产品形态”

更准确地说，现在的状态是：

- 所有子系统都已经进入代码骨架
- 最成熟的是同步、传输、元数据、存储主链
- 控制层和业务层已经有了清晰接口，下一步主要是往里面填真实业务逻辑

这个“同步内核”主要承担：

- 文件扫描
- 文件分块
- 差异比较
- 缺块请求
- 块传输
- 临时落盘
- 校验与原子提交
- 断点恢复

## 7. 当前代码架构说明

### 7.1 当前主入口

当前 1.0 的统一入口程序是：

- `apps/sync_v1/main.cpp`

它负责组织一次完整同步流程，包括：

- 启动服务端逻辑
- 启动客户端逻辑
- 服务端发送 `Hello`
- 服务端发送 manifest
- 客户端计算缺块请求
- 服务端返回 chunk 数据
- 客户端完成重建和提交
- 同步完成后调用存储后端

### 7.2 同步核心目录

当前代码骨架已经按现代 C++ 工程方式拆成：

- `apps/`：可执行程序入口
- `include/netdisk/`：公开头文件
- `src/`：实现代码
- `legacy_port/`：迁移过渡目录，当前运行主链已不再依赖
- `configs/`：运行配置
- `tests/`：测试预留目录

## 8. 核心模块说明

### 8.1 扫描模块

扫描模块负责遍历目录并生成文件记录。

主要职责：

- 递归遍历目录
- 只处理正式文件
- 忽略 `.part` 临时文件
- 把结果交给后续 chunker 和 metadata store

当前实现：

- `src/sync/scanner.cpp`

### 8.2 分块摘要模块

分块摘要模块负责把文件转成“适合同步的结构”。

主要职责：

- 按固定大小切块
- 计算每个块的 hash
- 计算整个文件的 root hash
- 生成 `FileRecord` 和 `ChunkRecord`

当前实现：

- `src/sync/chunker.cpp`
- `src/sync/hash.cpp`

当前哈希实现先使用 MD5，后续可以替换成更强的哈希算法。

### 8.3 差异比较模块

差异比较模块负责回答一个问题：

**本地到底缺什么？**

主要职责：

- 根据本地 SQLite 文件记录构建本地视图
- 接收远端 manifest
- 判断哪些文件完全一致
- 判断哪些文件缺失
- 判断哪些 chunk 需要拉取

当前实现：

- `src/sync/diff_engine.cpp`

### 8.4 传输协议模块

传输协议模块负责节点之间的消息交换。

当前消息类型包括：

- `Hello`
- `ManifestSummary`
- `MissingChunksRequest`
- `ChunkData`
- `SyncAck`
- `Error`

当前实现：

- `include/netdisk/network/protocol.h`
- `src/network/protocol_codec.cpp`
- `src/network/transport.cpp`

### 8.5 恢复写入模块

恢复写入模块负责把收到的数据安全地变成正式文件。

主要职责：

- 创建 `.part` 临时文件
- 支持从已有 `.part` 恢复
- 追加写入数据块
- 校验整文件 hash
- 校验通过后原子重命名为正式文件

当前实现：

- `src/sync/sync_writer.cpp`

### 8.6 元数据模块

元数据模块负责把同步过程中产生的状态记录下来。

当前保存的主要数据包括：

- `file_records`
- `chunk_records`
- `sync_tasks`
- `sync_task_chunks`
- `stored_objects`

主要作用：

- 记录本地有哪些文件
- 记录每个文件有哪些 chunk
- 记录任务进行到了哪一步
- 记录哪些 chunk 已完成
- 记录文件最终存到了哪里

当前实现：

- `src/metadata/metadata_store.cpp`

### 8.7 存储适配模块

存储适配模块负责把同步成功后的文件交给最终存储后端。

当前支持：

- `none`
- `mirror`
- `fastdfs`

设计思想是：

**同步逻辑和最终存储解耦。**

这样做以后，可以继续扩展新的存储后端，而不用改同步主链。

当前实现：

- `include/netdisk/storage/storage_backend.h`
- `src/storage/storage_backend.cpp`

## 9. 1.0 的核心数据流

当前 1.0 的一次同步，大致按下面流程执行：

1. 服务端扫描远端目录
2. 服务端生成 manifest
3. 客户端扫描本地目录
4. 客户端把远端 manifest 和本地元数据做比较
5. 客户端生成缺失文件和缺失块请求
6. 服务端按请求读取 chunk 并发送
7. 客户端把 chunk 写入 `.part`
8. 客户端完成整文件校验
9. 客户端原子提交正式文件
10. 客户端更新 SQLite 元数据
11. 客户端调用存储后端
12. 存储绑定结果写入 `stored_objects`

## 10. 分布式能力在系统中的位置

这个项目里的“分布式”不是单独一个页面功能，而是底层能力。

它主要落在两个位置：

- 子系统 2：同步与传输子系统
- 子系统 5：安全与可靠性子系统

更具体地说：

- 同步内核负责“文件怎么同步”
- 节点互联模块负责“节点怎么连上”

对于虚拟 IP / ZeroTier 这一层，当前建议采用：

- 第一阶段先复用官方客户端服务
- 后续再考虑用 `libzt` 把它集成到程序里

这样做的原因是：

- 当前最关键的是先把系统结构和同步逻辑做对
- 不要过早把底层网络集成复杂度引入主线

## 13. 1.0 在整个项目中的定位

当前 1.0 更适合被定义为：

**个人分布式网盘系统的同步内核原型版本。**

它不是整个最终系统，但它是最关键的底层核心组件之一。

你可以把整个项目理解成：

- 上层：业务和管理
- 中层：同步内核
- 下层：存储后端

其中：

- 业务层决定“同步什么”
- 同步内核决定“怎么安全同步”
- 存储层决定“最终存到哪里”

如果从用户视角重新描述它，可以写成：

**当前 1.0 已经先把“备份和传输”的底层发动机做出来了，后面要继续补的是“备份中心”和“共享文件库”这两个用户真正直接使用的业务壳层”。**

## 14. 下一阶段建议

在当前 1.0 基础上，建议按下面顺序继续推进：

1. 先把系统总体设计定稿
2. 明确控制面和业务层的数据模型
3. 补“设备管理子系统”
4. 补“备份中心子系统”
5. 补“中转共享库子系统”
6. 再逐步增强副本和容灾能力

也就是说，接下来不要只继续扩“同步内核”本身，
而是要开始把它放回整个系统结构中，作为一个核心组件来使用。
